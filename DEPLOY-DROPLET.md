# Deploying ThunderCitizen to a DigitalOcean Droplet

End-to-end runbook for self-hosting the full stack on a single VM using
`docker-compose.prod.yml`. This is the most reproducible path — anyone with
a domain, an SSH key, and ~$12/mo can stand up an identical instance.

```
   internet ──▶ caddy:443  ──▶ app:8080  ──▶ db:5432
                    │                          │
                    └─ Let's Encrypt cert      └─ ./secrets/postgres_password
                       (auto-renewed)             (generated by init)
```

## Prerequisites

- Registered domains — the compose file's Caddyfile currently serves `thundercitizen.ca` (canonical) and 302-redirects `thundercitizen.com` + `www.thundercitizen.com` to it. Edit `docker-compose.prod.yml`'s `configs.caddyfile.content` block if you want different hostnames.
- Access to that domain's DNS (registrar dashboard or DO domains)
- An SSH public key on your laptop (`~/.ssh/id_ed25519.pub` or similar)
- A DigitalOcean account

You do NOT need: doctl, a managed database, an object store, or anything else
DO-specific. The droplet is a plain Linux box; the stack is fully portable.

## 1. Create the droplet

DO control panel → Create → Droplets.

| Field | Value |
|---|---|
| Image | Ubuntu 24.04 LTS x64 |
| Region | Toronto (TOR1) — closest to Thunder Bay |
| Size | Basic / Regular SSD / **2 GB / 1 CPU** ($12/mo) |
| Authentication | SSH key — paste your public key |
| Hostname | `thundercitizen-prod` (or whatever) |

**Why 2 GB:** Postgres + PostGIS comfortably wants ~512 MB, the app ~100 MB,
caddy ~50 MB, plus OS overhead. The 1 GB plan technically works but leaves
no headroom for the GTFS-RT recorder's spikes. The 2 GB plan is the sweet
spot.

Note the public IPv4 once it's provisioned.

## 2. Point DNS at the droplet

At your registrar (or DO → Networking → Domains):

| Type | Name | Value | TTL |
|---|---|---|---|
| `A` | `@` (apex) | `<droplet IPv4>` | 300 |
| `A` | `www` | `<droplet IPv4>` | 300 |

(Add `AAAA` records too if you enabled IPv6 on the droplet.)

Verify before continuing — Let's Encrypt's HTTP-01 challenge in step 6
will fail if DNS hasn't propagated:

```bash
for d in thundercitizen.ca www.thundercitizen.ca thundercitizen.com www.thundercitizen.com; do
    printf '%-30s → %s\n' "$d" "$(dig +short "$d")"
done
# should return the droplet IP
```

## 3. SSH in and clone the repo

```bash
ssh root@<droplet IP>

mkdir -p /opt && cd /opt
apt-get update -qq && apt-get install -y -qq git
git clone --depth 1 https://github.com/eric-decompiled/ThunderCitizen.git
cd ThunderCitizen
```

## 4. Run the bootstrap script

One command. Idempotent. Installs ufw + fail2ban + unattended-upgrades,
configures the firewall (SSH/80/443 only), enables an sshd ban-after-5-fails
jail, turns on automatic security updates, and sets the timezone to UTC.

```bash
./scripts/harden.sh
```

See the comment header in the script for exactly what it does and what
it intentionally leaves alone (it doesn't touch sshd_config or create
non-root users — DO droplets created with key auth already have password
auth disabled).

## 5. Install Docker

The one-liner from docker.com installs `docker-ce` + the compose v2 plugin:

```bash
curl -fsSL https://get.docker.com | sh
docker --version          # → Docker version 27.x
docker compose version    # → Docker Compose version v2.x
```

## 6. (Optional) Override defaults

The prod compose file works with zero env vars — every value has a
sensible default:

| Var | Default |
|---|---|
| `ACME_EMAIL` | `eric@decompiled.dev` |
| `POSTGRES_USER` | `thundercitizen` |
| `POSTGRES_DB` | `thundercitizen` |
| `TC_IMAGE_TAG` | `latest` |

The canonical domain and the list of redirect aliases are hardcoded in the
Caddyfile content block of `docker-compose.prod.yml` rather than exposed
as env vars — changing them also means updating the `redir` target, which
is easier to get right by editing the file directly than by juggling env
vars.

To override anything, drop a `.env` file beside `docker-compose.prod.yml`:

```bash
cat > .env <<'EOF'
ACME_EMAIL=you@example.com
TC_IMAGE_TAG=sha-3d84894    # pin a specific build for reproducibility
EOF
```

The Postgres password is **auto-generated** on first boot by the `init`
service (64 hex chars from `/dev/urandom`) and stashed at
`./secrets/postgres_password`. You don't set it manually.

## 7. Bring up the stack

```bash
./scripts/run.sh
```

`run.sh` is a thin wrapper that saves typing `-f docker-compose.prod.yml`
on every command. With no args it runs `up -d`. Other useful forms:

| Command | What |
|---|---|
| `./scripts/run.sh` | bring the stack up |
| `./scripts/run.sh logs -f` | tail logs |
| `./scripts/run.sh ps` | service status |
| `./scripts/run.sh pull` | pull a fresh image, then up -d |
| `./scripts/run.sh down` | stop everything |

First boot order:
1. `init` runs once, generates `./secrets/postgres_password`, exits 0
2. `db` boots with that password as POSTGRES_PASSWORD_FILE, runs initdb
3. `app` reads the same secret file, builds DATABASE_URL, runs migrations
   on startup, then serves on `app:8080` inside the network
4. `caddy` reaches out to Let's Encrypt, completes the HTTP-01 challenge,
   provisions a cert for `$DOMAIN`, starts serving on 443

Watch it happen:

```bash
./scripts/run.sh logs -f
```

You should see caddy log a successful certificate obtain within ~30 seconds.

Verify from your laptop:

```bash
curl -I https://thundercitizen.ca/health
# and confirm the redirect works:
curl -I http://thundercitizen.com/   # should print HTTP/1.1 302 + Location: https://thundercitizen.ca/
# HTTP/2 200
```

## 8. Database seed (automatic)

Nothing to do here — both schema migrations and data patches (councillor
biographies, council motions/votes, FY2026 budget accounts + ledger, …)
are applied automatically by the app on every boot. Patches are
content-checksummed and tracked in `data_patch_log`, so subsequent boots
no-op once everything has been applied.

To verify after the first `./scripts/run.sh`:

```bash
./scripts/run.sh exec -T db psql -U thundercitizen -d thundercitizen -c \
    "SELECT patch_id, action, at FROM data_patch_log ORDER BY at;"
```

You should see one `apply` row per patch file in `patches/`.

To reseed from scratch (e.g. testing): wipe the postgres volume and bring
the stack back up — the app re-applies everything on boot.

```bash
./scripts/run.sh down
docker volume rm thundercitizen-prod_postgres_data
./scripts/run.sh up -d
```

## 9. Set up backups

`./scripts/backup.sh` is a thin `pg_dump` wrapper that writes
gzipped dumps to `./backups/` via the bind mount on the `db` service.
Filenames are `thundercitizen-<UTC ISO timestamp>.sql.gz` so they sort
chronologically.

One-shot test:

```bash
./scripts/backup.sh
ls -la backups/
```

Daily cron at 04:00 UTC:

```bash
crontab -e
```

```cron
0 4 * * * cd /opt/ThunderCitizen && ./scripts/backup.sh >> /var/log/tc-backup.log 2>&1
```

Off-host shipping (rclone to S3/B2, restic, etc.) and retention pruning are
intentionally **not** in the script — wire those up to whatever you already
use for off-host storage. A trivial retention example:

```bash
# Keep 30 days of dumps
find /opt/ThunderCitizen/backups -name 'thundercitizen-*.sql.gz' -mtime +30 -delete
```

## 10. Updates

The CI workflow publishes a fresh image to GHCR on every push to `main`.
To roll forward:

```bash
cd /opt/ThunderCitizen
git pull
docker compose -f docker-compose.prod.yml pull app
docker compose -f docker-compose.prod.yml up -d app
```

If a release modifies the compose file itself or migrations, the `git pull`
+ `up -d` covers it. The `db` and `caddy` services are unaffected by app
updates and stay running.

To pin to a specific build (recommended for production):

```bash
echo "TC_IMAGE_TAG=sha-$(curl -s https://api.github.com/repos/eric-decompiled/ThunderCitizen/commits/main | grep -o '"sha":"[^"]*"' | head -1 | cut -d'"' -f4 | cut -c1-7)" >> .env
docker compose -f docker-compose.prod.yml up -d app
```

## 11. Restoring from a backup

```bash
# Stop the app so nothing writes to the DB during restore
docker compose -f docker-compose.prod.yml stop app

# Pipe a dump back into psql
gunzip -c backups/thundercitizen-2026-04-10T12-00-00Z.sql.gz | \
  docker compose -f docker-compose.prod.yml exec -T db \
  psql -U thundercitizen -d thundercitizen

# Bring the app back
docker compose -f docker-compose.prod.yml start app
```

For a full disaster recovery (fresh droplet, no existing data):
provision per steps 1-5, copy a recent dump to `./backups/`, then
follow the restore command above before bringing up the app. The
`init` service will still generate a fresh password — the dump's
data overwrites whatever the fresh DB started with.

## Troubleshooting

**Caddy can't get a cert:**
- DNS hasn't propagated yet → wait, then `docker compose restart caddy`
- Port 80 isn't reachable → check `ufw status` and DO's cloud firewall
- Rate-limited by Let's Encrypt (if you bounced the stack a lot in testing)
  → wait an hour, or use the staging directory for testing

**App can't connect to DB:**
```bash
docker compose -f docker-compose.prod.yml logs app | tail -30
docker compose -f docker-compose.prod.yml exec db psql -U thundercitizen -l
```

**Force GTFS reload** (routes disappeared but hash still stored):
```bash
./scripts/run.sh exec -T db psql -U thundercitizen -d thundercitizen \
    -c "DELETE FROM transit_feed_state WHERE feed_type = 'gtfs_static';"
./scripts/run.sh up -d --force-recreate app
./scripts/run.sh logs --since=30s -f app 2>&1 | grep -i gtfs
```

**Disk filling up:**
```bash
docker system df          # what's eating space
docker image prune        # old images
du -sh backups/           # backup retention
du -sh /var/lib/docker/volumes/thundercitizen-prod_postgres_data
```

**Reset the password (lost it / leaked it):**
```bash
docker compose -f docker-compose.prod.yml down
rm secrets/postgres_password
docker volume rm thundercitizen-prod_postgres_data   # WIPES DATA
docker compose -f docker-compose.prod.yml up -d
# then re-apply patches and restore the latest backup
```

**Where things live on the droplet:**

| Path | What |
|---|---|
| `/opt/ThunderCitizen/` | the cloned repo |
| `/opt/ThunderCitizen/secrets/postgres_password` | DB password (chmod 644) |
| `/opt/ThunderCitizen/backups/` | dumps from `backup.sh` |
| `/var/lib/docker/volumes/thundercitizen-prod_postgres_data/` | actual DB files |
| `/var/lib/docker/volumes/thundercitizen-prod_caddy_data/` | TLS certs |
